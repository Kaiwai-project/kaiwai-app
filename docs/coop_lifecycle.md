# 카이와이 공구방 라이프사이클 및 노출/삭제(상태 관리) 요구사항

공구방(Coop Room)의 상태(status) 변화와 그에 따른 노출 범위, 그리고 삭제 시점을 다음의 룰에 따라 구현합니다. DB의 레코드를 실제로 지우는 Hard Delete는 최소화하고, 상태값을 변경하여 UI 노출을 제어하는 Soft Delete를 기본 원칙으로 합니다.

## 1. 상태별 노출 범위 (Visibility)
- `recruiting` (모집중): 누구나 메인 피드 및 검색에서 볼 수 있음.
- `closed` (모집완료/출발): 
  - 트리거: 목표 인원 달성 또는 호스트의 수동 마감 클릭
  - 액션: 즉시 메인 피드 및 일반 검색 목록에서 감춤.
  - 노출: 오직 호스트와 탑승자(참여자)의 '내 참여 내역/관리 페이지'에서만 접근 가능.
- `completed` (수령완료/종료): 호스트가 물품 분배 완료를 선언한 상태. 당사자들의 '과거 내역' 탭으로 이동됨.
- `canceled` (취소됨): 피드 노출 즉시 차단, 당사자 내역에만 표시.

## 2. '삭제' 타이밍 및 처리 방식 (Lifecycle End)
다음 조건에 따라 공구방의 최종 처리(삭제 및 숨김)를 진행합니다.

1. [즉시 완전 삭제]: 
   - 상태: `recruiting`
   - 조건: 탑승자가 '0명'인 상태에서 호스트가 방 삭제를 요청할 경우.
   - 처리: DB에서 즉시 삭제 (Hard Delete).

2. [안전 보관 및 UI 만료 (Soft Delete)]:
   - 상태: `completed` (수령 완료)
   - 조건: '수령 완료' 상태로 변경된 지 30일(또는 14일)이 경과함.
   - 처리: 유저들의 내역 목록에서 기본적으로 숨김 처리(Archived). 단, DB에는 거래 내역 증빙을 위해 최소 1년간 보존. (유저가 직접 '내역에서 삭제' 버튼을 누르면 즉시 UI에서 숨김)

3. [자동 취소 및 만료 처리]:
   - 상태: `recruiting`
   - 조건: 설정된 모집 마감 기한이 지났으나 목표를 채우지 못함.
   - 처리: 상태를 `expired`(또는 canceled)로 변경. 탑승자가 있다면 전액 포인트 자동 환불. 7일 후 당사자들의 내역에서도 자동 숨김.

4. [분쟁 중 보존 잠금 (Lock)]:
   - 조건: 해당 공구방에 대해 탑승자로부터 '미수령/사기' 신고가 접수됨.
   - 처리: 관리자(Admin)의 개입 전까지 어떠한 자동 삭제/만료/숨김 로직도 동작하지 않도록 예외 처리함.

## 3. 필요 작업
- Supabase의 공구방 테이블(`coop_rooms`)에 상태 관리를 위한 칼럼 추가/확인 (`status` enum, `expires_at`, `completed_at` 등)
- 메인 피드 데이터를 불러오는 RPC/쿼리에서 `status = 'recruiting'`만 가져오도록 필터링 필수 적용.
- (선택) Supabase pg_cron이나 Edge Functions의 스케줄러를 이용해 매일 자정 만료된 방의 상태를 업데이트하는 배치 작업 구현.
